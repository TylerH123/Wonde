# Space scaffold

Recommended workflow for a new space:

```bash
spaces create-action --space wonde-doors --name 'Hello World'
spaces build-actions --space wonde-doors
spaces verify-space --space wonde-doors
spaces build-space --space wonde-doors
# edit src/App.tsx to import { Space } from "./actions"
# App.tsx MUST keep <SpaceRoot> from @hatch/sdk/components as the outermost wrapper
spaces create-icon --space wonde-doors --prompt '...'
```

Available imports by default:
- `react`
- `@hatch/sdk`
- `@hatch/sdk/components`
- local relative modules inside this space directory

The exported app MUST start with `SpaceRoot` from `@hatch/sdk/components` as its
outermost wrapper. It owns the host header inset (`--header-height`), optional
host bottom inset (`--bottom-padding`), viewport height, and safe-area bottom
padding contract for spaces. Do not recreate those with `body` styles or ad-hoc
top padding in the space itself. `SpaceRoot` already paints `--bg` by
default. If you need a non-default full-bleed app background such as a gradient or
image, set `backgroundColor`, `background`, or `backgroundImage` on
`SpaceRoot` itself so it extends behind the host header and safe area.

The scaffold includes `src/theme.css` with local `--bg`, `--surface`, `--text`,
`--dim`, `--border`, `--accent`, and `--radius` tokens. Keep every space UI readable
in both light and dark mode by setting `color-scheme: light dark` and using
`prefers-color-scheme` overrides in that local theme layer. `src/main.tsx`
imports that theme by default, so scaffold rebuilds keep the base theme wiring
intact even if `src/App.tsx` is rewritten. The scaffolded `tailwind.config.cjs`
also owns the default type scale for the space, so adjust typography
there or in local CSS rather than expecting defaults from `@hatch/sdk`. Spaces do not ship
a Hatch theme SDK layer.

Actions live in `actions/` and should use Pydantic models with handlers like `async def main(ctx, request: Request)`.
The framework parses the request for you. After changing actions, run
`spaces build-actions --space wonde-doors` to typecheck them and regenerate `src/actions.ts`.
Frontend code should import `Space` from `src/actions.ts` and call typed helpers like
`await Space.helloWorld(request)` instead of raw string action names.
`src/actions.ts` is generated by the spaces CLI and should not be edited by hand.
For workspace debugability, keep action-level logs in
`$JARVIS_HOME/workspace/spaces/wonde-doors/logs/space-debug.log` as JSONL.
The scaffolded action templates include best-effort file logging helpers. Route
important frontend events/errors through a dedicated logging action (for
example `log_client_event`) so app-side behavior lands in the same file.
The canonical debug log is reset once per UTC day to prevent unbounded growth.
Before publishing, run `spaces verify-space --space wonde-doors`. It runs smoke tests from
`actions/tests/` and probes the real cache-backed read actions against `app.db`. For actions that
need required args or do not use `get_` / `list_` / `fetch_` style names, declare them explicitly
under `verification.criticalActions` in `space.json`.
`spaces create-action` also scaffolds a matching smoke test in `actions/tests/`; extend that file
instead of replacing it with a custom runner.
The scaffold also includes `verify/playwright.verify.mjs`. Tailor it to the space's critical user
journey instead of deleting it. `space.json` defaults Playwright to hybrid mode so browser action
requests fall back to the real Python runtime when the verifier hook does not return custom data.
Smoke tests for Python actions live under `actions/tests/` and should be run from the
`actions/` directory with the shared venv, for example:

```bash
cd "$JARVIS_HOME/workspace/spaces/wonde-doors/actions"
"$JARVIS_HOME/workspace/spaces/.venv/bin/python3" -m unittest discover -s tests -p 'test_*.py'
```

Smoke tests should use ordinary imports from `actions/`, such as
`from list_todos import main, Request`. They must be discoverable by that `unittest` command:
use `unittest.TestCase` methods named `test_*` or a module-level `load_tests(...)`.
Custom `if __name__ == "__main__"` smoke runners are not executed by `build-space` or
`verify-space`. Do not modify `sys.path` in smoke tests.
Per-space SQLite data lives in `app.db`.
Python action dependencies live in `actions/requirements.txt` and are installed into the shared
`workspace/spaces/.venv` environment when you run `spaces build-actions --space ...`.
That file should only list space-local extra packages; the managed baseline is provisioned for you.
Frontend dependencies live in `package.json` at the space root and are installed into the shared
workspace Bun store under `workspace/spaces` when you run `spaces build-space --space ...`.
`react`, `react-dom`, `lucide-react`, `recharts`, Tailwind, and a managed vendored `@hatch/sdk`
tarball dependency are scaffolded for you.
Spaces use Bun-based `index.html` + `src/main.tsx` bootstrapping, while your app entry stays in `space.json`.
The vendored `@hatch/sdk` tarball is managed by Jarvis and should not be edited directly.
Generate local icon assets with `spaces create-icon --space ... --prompt '...'`; it will save `icon.jpg` and derive `icon.png`.
